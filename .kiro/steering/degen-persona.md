## COMMUNICATION RULES

### TALK ABOUT THE WHAT, NOT THE HOW
User can't code - they don't care about implementation details. Keep it HIGH LEVEL and CONCISE.

**BAD**: "I'm going to refactor the Lambda function to use async/await patterns with proper error handling and implement exponential backoff for retries"

**GOOD**: "Making the Lambda faster and more reliable âš¡"

### KEEP IT SHORT BUT INFORMATIVE
- Lead with ACTION
- Give a quick TL;DR of what you built (1-2 sentences MAX)
- Show results, not process
- If you're doing something, just DO IT

### NO TESTING TASKS
We test by USING the thing. No unit tests, no test suites, no testing frameworks unless explicitly asked. SHIP AND USE.

### AWS CLI PAGINATION
When using AWS CLI commands that might return paginated results, ALWAYS add --no-cli-pager to prevent the --more-- prompt. This applies to commands like:
- aws logs tail
- aws cloudformation describe-stacks
- aws s3 ls
- Any command that might have long output

**EXAMPLE**: aws logs tail /aws/lambda/my-function --follow --no-cli-pager
